# OD 단독 검증 — Design EC2 서버 runbook

Teamver design-api · project registry · S3 access gate **없이** Open Design(daemon) 코어(채팅, artifact, **슬라이드/deck**)가 정상인지 확인하는 절차.

**언제 쓰나**

- `/access` 404, registry, 세션 이슈와 **슬라이드 생성 실패**를 분리하고 싶을 때
- API 키(BYOK)는 서버에 넣어뒀는데 **OD 엔진 자체**가 깨졌는지 볼 때
- Teamver 연동 배포 전·후 **회귀 기준선**을 잡을 때

**관련 스크립트**

| 스크립트 | 역할 |
|----------|------|
| `deploy/teamver/scripts/run_od_core_verify.sh` | `:7457` 격리 스택 기동/종료 + full 검증 |
| `deploy/teamver/scripts/verify_od_core.sh` | daemon HTTP 스모크 (health, deck skill, BYOK) |
| `deploy/teamver/scripts/seed_od_byok_app_config.sh` | `app-config.json` BYOK 주입 |
| `deploy/teamver/scripts/seed_od_runtime_config.sh` | onboardingCompleted seed |
| `deploy/teamver/docker-compose.od-core-verify.yml` | 격리 compose (Teamver gate OFF) |

**대조:** Teamver 연동 전체 스모크 → `bash scripts/smoke_design.sh --staging`

---

## 0. 전제

| 항목 | Staging Design EC2 |
|------|-------------------|
| 작업 디렉터리 | `/opt/teamver-design` (또는 clone 경로의 `deploy/teamver/`) |
| 기존 sidecar | `127.0.0.1:7456` (nginx → `stg-design.teamver.com`) |
| 격리 검증 스택 | `127.0.0.1:7457` (**기존 7456 중단 없음**) |
| env 파일 | `.env.staging` — **git commit 금지** (`.env` symlink 불필요, compose 가 `--env-file` 로 직접 읽음) |
| LLM 키 | `.env.staging` 의 `ANTHROPIC_API_KEY` (또는 `OPENAI_API_KEY`) |

**중요:** 현재 Docker 이미지는 `VITE_TEAMVER_EMBED=1` 빌드라 UI는 Teamver Design 브랜딩·embed 제약( Settings execution 숨김 등)이 남습니다.
**OD 코어 검증**은 daemon API + BYOK `app-config` 로 가능합니다. 순수 upstream OD UI 가 필요하면 소스 dev 빌드(embed flag OFF)를 별도 runbook 으로 진행하세요.

## 1. EC2 접속

```bash
ssh -i ~/.ssh/your-key.pem ubuntu@<design-staging-ec2>
cd /opt/teamver-design   # deploy/teamver 루트
git pull origin staging  # 최신 스크립트 반영
```

---

## 2. env 확인 (API 키 · OD 토큰)

```bash
cd /opt/teamver-design
grep -E '^(OD_API_TOKEN|ANTHROPIC_API_KEY|OPENAI_API_KEY|OPEN_DESIGN_IMAGE)=' .env.staging \
  | sed 's/=.*/=***/'
```

| 변수 | 필수 | 설명 |
|------|------|------|
| `OD_API_TOKEN` | ✓ | daemon Bearer (nginx·loopback 외 API) |
| `ANTHROPIC_API_KEY` | ✓* | BYOK chat (* 또는 OPENAI) |
| `OPEN_DESIGN_IMAGE` | 권장 | 배포 중인 OD 이미지 tag |

키가 없으면 `.env.staging` 에 추가 후 **격리 스택만** 재기동 (§3).

---

## 3. 방법 A — 격리 스택 (권장, 7456 무중단)

Teamver design-api / registry / `/access` gate **완전 OFF**, 로컬 디스크 storage.

### 3.1 한 번에 (start + BYOK seed + curl 검증)

```bash
cd /opt/teamver-design
bash scripts/run_od_core_verify.sh full --staging
```

성공 시 마지막에 `✓ verify_od_core OK`.

### 3.2 단계별

```bash
# 1) 기동 (:7457)
bash scripts/run_od_core_verify.sh start --staging

# 2) onboarding + BYOK app-config
bash scripts/run_od_core_verify.sh seed --staging

# 3) HTTP 스모크 (+ 선택: 프로젝트 create/list)
bash scripts/run_od_core_verify.sh verify --staging
bash scripts/run_od_core_verify.sh verify --staging --project-smoke

# 4) 종료 (데이터 volume 유지)
bash scripts/run_od_core_verify.sh stop --staging
```

### 3.3 격리 스택에서 확인되는 것

- `GET /api/health` 200
- `GET /api/skills` 에 deck skill (`simple-deck`, `magazine-web-ppt`)
- `TEAMVER_DESIGN_API_URL` **비어 있음** → project access middleware 비활성
- `OD_PROJECT_STORAGE=local` → S3 tenant gate 없음
- `app-config` `mode=api` + apiKey 설정됨

---

## 4. 방법 B — 기존 sidecar(7456) 상태만 점검

Teamver 연동 **켜진 채** daemon API 가 살아있는지 빠르게 볼 때.

```bash
cd /opt/teamver-design
bash scripts/verify_od_core.sh --staging --expect-teamver-gate
```

| 결과 | 의미 |
|------|------|
| health/skills OK, gate ON | daemon 살아 있음. deck/chat 문제는 registry·세션·S3 쪽 가능 |
| health FAIL | daemon/nginx/compose 장애 |
| gate OFF 인데 7456 사용 중 | `.env.staging` 의 `TEAMVER_DESIGN_API_URL` 확인 |

BYOK seed (기존 daemon):

```bash
bash scripts/seed_od_runtime_config.sh --staging
bash scripts/seed_od_byok_app_config.sh --staging
bash scripts/verify_od_core.sh --staging --expect-teamver-gate
```

## 5. 브라우저 — Deck(슬라이드) 수동 검증

curl 로는 **LLM 생성·artifact 렌더**까지 검증하지 않습니다. 아래를 **격리 스택(7457)** 또는 **SSH 터널** 로 진행합니다.

### 5.1 SSH 터널 (로컬 PC → EC2 :7457)

로컬 터미널:

```bash
ssh -L 7457:127.0.0.1:7457 ubuntu@<design-staging-ec2>
```

브라우저: `http://127.0.0.1:7457`

> 격리 스택이 `run_od_core_verify.sh start` 로 떠 있어야 합니다.

### 5.2 Staging 도메인 (7456, Teamver nginx)

`stg-design.teamver.com` — nginx 로그인 게이트 + Cookie SSO.
Teamver **연동 포함** E2E. OD-only 와 혼동하지 말 것.

nginx·토큰 주입: `deploy/teamver/devops/nginx/README.md`

### 5.3 Deck 수동 체크리스트

1. **New project** → 탭 **Slide deck**
2. **Skill:** `simple-deck` (가벼움) 또는 `magazine-web-ppt` (기본 deck)
3. **Design system:** `Neutral Modern` 등
4. Execution: **API mode** (BYOK seed 후)
5. 프롬프트 예:
   ```text
   Create a 5-slide pitch deck about AI design tools. Include speaker notes.
   ```
6. **성공 기준**
   - [ ] 채팅 스트리밍 완료
   - [ ] 오른쪽 preview 에 HTML deck
   - [ ] 슬라이드 2장 이상 + prev/next 또는 스와이프
   - [ ] Project files 에 `.html` 저장

### 5.4 실패 시 구분

| 증상 | OD core | Teamver 연동 |
|------|---------|--------------|
| Send 직후 401/502 | API key / app-config | runtime-config / design-api |
| 스트리밍 OK, artifact 없음 | skill·provider | — |
| deck HTML OK, 페이지 네비 깨짐 | OD preview | — |
| 프로젝트 열기 404 | — | registry / `/access` |
| `project_not_found` 로그 | — | design-api + daemon gate |

## 6. (선택) loopback API 로 프로젝트·chat 사전 점검

EC2 **내부**에서 (loopback 은 Bearer 생략 가능):

```bash
# health
curl -sS http://127.0.0.1:7457/api/health | head

# deck skill 목록
curl -sS http://127.0.0.1:7457/api/skills | grep -o 'simple-deck\|magazine-web-ppt' | head

# app-config (키 마스킹 — 컨테이너 내부)
docker exec teamver-od-core-verify node -e "
const fs=require('fs');
const p='/app/.od/app-config.json';
const j=JSON.parse(fs.readFileSync(p,'utf8'));
console.log({mode:j.mode, protocol:j.apiProtocol, hasKey:!!j.apiKey, model:j.model});
"
```

실제 chat/stream 은 브라우저 또는 Playwright e2e (`e2e/ui/app.test.ts` deck 시나리오) 권장.
