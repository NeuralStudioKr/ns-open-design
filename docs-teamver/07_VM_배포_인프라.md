## 한 줄 결론

> **Staging EC2와 Production EC2를 분리**하고, slides/docs 와 같이 **UI·API 서브도메인 2개**로 노출한다.  
> 인프라는 **AWS EC2 + RDS** (staging: BE RDS 공유 / prod: 전용 RDS), Production TLS는 **ALB(ACM)**, Staging TLS는 **EC2 Nginx + certbot**.

## 1. 왜 호스트를 나누나

| 결정 | 내용 |
|------|------|
| **stg ≠ prod EC2** | 데이터·토큰·장애 격리. slides(`stg-slides*`)·docs(`docs*`) 와 동일 원칙 |
| **1 EC2에 stg+prod 공존 안 함** | 리소스 경합·설정 실수·OD_DATA_DIR 혼선 위험 |
| **Main BE는 또 다른 호스트** | `stg-api` / `api.teamver.com` — Design EC2 와 분리 (기존 Teamver 인프라) |
| **DB는 compose Postgres 아님** | **Design sidecar 전용 AWS RDS** — staging: `teamver-staging-postgres` + `teamver_design_staging` / prod: `teamver-design-prod-postgres` + `teamver_design_production` |
| **Main BE production DB** | **GCP Cloud SQL** (`api.teamver.com`) — Design EC2 `POSTGRES_*` 와 **무관** |

### 다른 AI Apps 와의 대응

| 앱 | Staging | Production | 패턴 |
|----|---------|------------|------|
| Slides | `stg-slides` / `stg-slides-api` | `slides` / `slides-api` | AI Apps Staging VM · prod 별도 |
| Docs | (통합 prod VM) | `docs` / `docs-api` | FE+BE 서브도메인 |
| **Design** | `stg-design` / `stg-design-api` | `design` / `design-api` | **동형** + OD sidecar |

Design 만 OD daemon(`:7456`)이 UI+API를 한 포트에서 서빙한다. Teamver 측은 여전히 **UI 도메인 + API 도메인** 2개를 둔다.

## 2. 전체 토폴로지

```text
                    ┌─────────────────────────────────────┐
                    │  Main BE (기존)                       │
                    │  stg-api.teamver.com / api.teamver.com│
                    └──────────────▲──────────────────────┘
                                   │ bootstrap · session-check
┌──────────────────────────────────┼──────────────────────────────────┐
│ Design Staging EC2 (AWS)         │ Design Production EC2 (AWS)       │
│                                  │                                  │
│ stg-design.teamver.com ──► :7456 │ design.teamver.com ──► :7456     │
│   (OD web + daemon)              │   (OD web + daemon)              │
│ stg-design-api ──────────► :16000│ design-api ──────────► :16000    │
│   (teamver-design-api)           │   (teamver-design-api)           │
│                                  │                                  │
│ TLS: EC2 Nginx + Let's Encrypt   │ TLS: ALB (ACM) → EC2 nginx :80   │
│ DNS(사용자): EIP A 레코드         │ DNS(사용자): ALB CNAME           │
│ EIP: SSH·배포 (동일)              │ EIP: SSH·배포 (동일, 브라우저 DNS 아님) │
└──────────────────────────────────┴──────────────────────────────────┘
         │                                    │
         ▼                                    ▼
   teamver-staging-postgres          teamver-design-prod-postgres
   DB: teamver_design_staging        DB: teamver_design_production
```

**브라우저 흐름**

1. Main FE → AI Apps → Design (Cookie SSO refresh)
2. `stg-design` / `design` — OD UI (nginx 로그인 게이트 + `OD_API_TOKEN` 주입)
3. OD web → `stg-design-api` / `design-api` — SSO·bootstrap·usage

## 3. 호스트별 구성

### 3.1 Design Staging EC2

| 항목 | 값 |
|------|-----|
| **AWS 타입 (권장)** | `t3.large` (2 vCPU, **8 GiB**) — OD + design-api + export 여유 |
| **디스크** | root 50GB + EBS data 30GB (`OD_DATA_DIR`) — §3.5 |
| **EIP** | **있음** — **사용자 DNS(A 레코드) + SSH·배포** (`terraform output ssh_command` / `ec2_public_ip`) |
| **DNS (사용자 트래픽)** | `stg-design.teamver.com`, `stg-design-api.teamver.com` → **EIP** (Terraform Route53 또는 수동 A) |
| **env** | `.env.staging` — `POSTGRES_HOST` = teamver-staging RDS endpoint |
| **S3 creds** | EC2 instance profile `teamver-design-staging-app` + IMDS hop **2** (권장). 임시 static key 가능 — [18 Instance Profile](./18_EC2_IAM_Instance_Profile_S3_설정.md) |
| **기동** | `bash deploy.sh --staging --rds` |

**컨테이너** (RDS 사용 시 `design-db` 없음)

| 서비스 | 호스트 포트 | 비고 |
|--------|-------------|------|
| `open-design-daemon` | 7456 | loopback only |
| `teamver-design-api` | 16000 | loopback only |

**Nginx:** `stg-design.teamver.com.{http,https}.conf` — TLS 종료, `auth_request` → `stg-api.teamver.com`

### 3.2 Design Production EC2

| 항목 | 값 |
|------|-----|
| **AWS 타입 (권장)** | **`t3.xlarge`** (4 vCPU, **16 GiB**) — staging `t3.large` 대비 상향 |
| **디스크** | root **100GB** + EBS data **100GB** (`OD_DATA_DIR`) |
| **EIP** | **있음** (`enable_ec2_eip=true`) — **SSH·배포·운영** (`terraform output ssh_command` / `ec2_public_ip`). SSM 가능 (`ssm_command`) |
| **DNS (사용자 트래픽)** | `design.teamver.com`, `design-api.teamver.com` → **ALB** (GCP CNAME). **EIP로 A 레코드 하지 않음** — staging 과 달리 브라우저는 EIP를 거치지 않음 |
| **env** | `.env.production` — S3 access key **금지**, instance profile only |
| **S3 creds** | `teamver-design-prod-app` + IMDS hop **2** — [18 Instance Profile](./18_EC2_IAM_Instance_Profile_S3_설정.md) |
| **기동** | `bash deploy.sh --production --rds` |

**Nginx:** `design.teamver.com.http.conf` only — **EC2 443/Let's Encrypt 사용 안 함** (ALB가 TLS 종료)

**ALB**

- `idle_timeout`: **3600s** (OD SSE)
- 헬스: `/_nginx/health`
- 대상: EC2 nginx **:80** (Host 기반 라우팅은 nginx)

### 3.3 Main BE (기존 — Design 팀이 새로 띄우지 않음)

| 환경 | 호스트 | DB | Design 에서 쓰는 API |
|------|--------|-----|----------------------|
| Staging | `stg-api.teamver.com` | AWS RDS (`teamver-staging`) | `GET /api/auth/session-check` (nginx) |
| Production | `api.teamver.com` | **GCP Cloud SQL** | 동일 (HTTP만 — Design DB 아님) |

### 3.4 Design sidecar RDS (teamver-design-api schema)

**Main BE DB와 분리.** Terraform SSOT: `ns-teamver-devops/.../teamver-design/docs/DATABASE.md`

| 환경 | RDS 인스턴스 | Database | user |
|------|--------------|----------|------|
| Staging | **`teamver-staging-postgres`** (BE와 동일) | `teamver_design_staging` | `teamver_be_admin` |
| Production | `teamver-design-prod-postgres` (**AI Design 전용**, Terraform) | `teamver_design_production` | `teamver_design_admin` |

호스트: `terraform output postgres_host`
Staging: `CREATE DATABASE` 1회 — `terraform output rds_create_database_sql`
Production: Terraform이 `db_name` 생성
`POSTGRES_SSLMODE=require` 권장 (AWS RDS).

### 3.5 EC2 EBS 볼륨 (root + od-data)

**Staging·Production 모두 EBS 2개.** Terraform `ebs_data.tf` — `od_data_volume_gb > 0` 이면 root 외 **별도 data 볼륨** 1개를 `/dev/sdf`로 부착한다 (`od_data_volume_gb = 0` 이면 root 아래 `od-data`만 사용).

| 볼륨 | 디바이스 (일반적) | 마운트 | 용도 |
|------|-------------------|--------|------|
| **Root** | `/dev/nvme0n1` | `/` | Ubuntu, Docker 이미지·레이어, `/opt/teamver-design` 배포·compose, nginx |
| **OD data** | `/dev/nvme1n1` | `/opt/teamver-design/od-data` | `OD_DATA_DIR` — `app.sqlite`, scratch, export 캐시, Litestream replica 경로 등 |

부트스트랩(`templates/ec2_bootstrap.sh.tpl`)이 data 볼륨을 ext4 포맷·`/etc/fstab` 등록·`ubuntu` 소유로 마운트한다. `/etc/teamver-design.env` 의 `TEAMVER_DESIGN_OD_DATA_DIR` 에 경로가 기록된다.

**환경별 용량** (`staging.terraform.tfvars` / `prod.terraform.tfvars`):

| 환경 | Root | OD data EBS | Terraform 리소스명 |
|------|------|-------------|-------------------|
| Staging | 50 GiB | 30 GiB | `teamver-design-staging-od-data` |
| Production | 100 GiB | 100 GiB | `teamver-design-prod-od-data` |

Production이 더 큰 이유: export·동시 사용자·Docker 여유 (`t3.xlarge` 상향과 동일 맥락). **볼륨 개수 차이가 아니라 용량 차이**이다.

**역할 (저장소 SSOT):** 사용자 프로젝트 파일 SSOT는 **S3** (`OD_PROJECT_STORAGE=s3`). OD data EBS는 **scratch/cache·SQLite·로컬 메타** 용 — [09 저장소·격리](./09_Design_저장소_격리_출시게이트.md), **상세:** [20 Hybrid 저장소 가이드](./20_Design_Hybrid_저장소_로컬_S3_가이드.md). root 가 꽉 차지 않도록 data I/O 를 분리한다.

**현장 확인 (SSH):**

```bash
lsblk
df -h / /opt/teamver-design/od-data
```

`nvme1n1` 이 없고 `od-data` 가 root 와 같은 디스크에만 있으면 → 예전 apply( data 볼륨 미부착 ) 또는 `od_data_volume_gb = 0` 시점 인스턴스. `teamver-design` terraform re-apply 로 data EBS 부착.

## 4. 리소스 튜닝

OD EC2는 **LLM 서버가 아니다**. daemon은 오케스트레이션·export·파일 I/O; 모델 호출은 BYOK/외부 API.

| 구간 | Staging | Production |
|------|---------|------------|
| `OD_MEM_LIMIT` | `1536m` | `2g` |
| `NODE_OPTIONS` | `--max-old-space-size=1024` | `--max-old-space-size=1536` |
| export 스파이크 | Chrome subprocess +500MB~1GB 일시 | 동일 |

compose 기본 `512m` 은 **로컬 검증용**. Staging/Production EC2 에서는 위 값 사용.

**디스크:** §3.5 — root + od-data **2볼륨**. od-data 는 **scratch/cache 전용** — [20 Hybrid 저장소](./20_Design_Hybrid_저장소_로컬_S3_가이드.md) §4 (evict·알람).  
용량 모니터링 80% 초과 시 EBS 확장(`od_data_volume_gb` / root `ec2_root_volume_gb`) 또는 scratch 정리.

| 파일 | 호스트 | 주요 차이 |
|------|-----|-----------|
| `.env.staging.example` | Staging | `POSTGRES_HOST=<RDS endpoint>`, `POSTGRES_DB=teamver_design_staging` |
| `.env.production.example` | Production | `POSTGRES_HOST=<RDS endpoint>`, `POSTGRES_DB=teamver_design_production` |

**서버에만 두는 비밀 (git 제외)**

- `OD_API_TOKEN` — nginx `teamver-design-od-token.conf` 와 compose `.env` **동일 값**
- `TEAMVER_JWT_SECRET`, `TEAMVER_INTERNAL_API_KEY` — Main BE 와 동기화
- `POSTGRES_PASSWD` — staging: teamver-staging RDS master; prod: **Design prod RDS master** (`TF_VAR_teamver_design_rds_pass`, user `teamver_design_admin`) — **Main BE GCP DB 비밀번호 아님**
