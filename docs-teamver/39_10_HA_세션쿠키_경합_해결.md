## 1. 증상 (운영에서 본 모습)

### 1.1 HTTP 응답

```json
{
  "detail": "session_expired",
  "login_url": "https://stg.teamver.com/auth/signin?app_id=teamver-design&redirect_url=https%3A%2F%2Fstg-design.teamver.com%2Fauth%2Fcallback"
}
```

- 경로 예: `/teamver-bff/drive/api/drive/folder`, `/teamver-bff/drive/api/drive/list`, `/teamver-bff/drive/api/v2/shared-drive`
- 의미: Main Drive가 “폴더 ACL로 거절”한 것이 아니라, **design-api가 현재 요청의 BFF session(access/refresh)을 신뢰할 수 없다**고 판단한 것.

### 1.2 UX 패턴

1. 재로그인 직후 수 분~수 요청은 정상.
2. embed를 쓰면서 `/auth/session`, workspace, bootstrap, Drive 모달 등 **여러 API가 병렬**로 나가면 어느 순간부터 Drive만(또는 Drive 포함해) 401 폭주.
3. 로그아웃/재로그인하면 잠시 회복 → 같은 패턴 반복.
4. **ACTIVE 노드 1대만** target group에 두면 재현이 사라지거나 급감한다.

### 1.3 유사 증상과 구분

이중화 전후로 Drive 401은 **여러 층**이 겹친다. 진단 순서를 헷갈리지 말 것.

| 층 | 문서 / 커밋 계열 | 대표 응답 | 핵심 |
|----|------------------|-----------|------|
| A. nginx `auth_request`가 Drive보다 먼저 refresh | [22 §2.1b](./22_Drive_인증_Usage_연동_검토.md) · 2026-07-13 | Main 원문 `Invalid token` | `/teamver-bff/drive/`는 auth_request **제외**, router가 refresh 소유 |
| B. multi-node refresh 후 **stale Set-Cookie** (본 문서) | 본 문서 · `9c4c266c9` | BFF `session_expired` | **모든 응답 Set-Cookie** + rotation 쿠키 롤백 |
| C. Design/Main 쿠키명 `session` 충돌 | [22 §3.2e](./22_Drive_인증_Usage_연동_검토.md) · `a88e0d622` | 빈 session → `session_expired` | 쿠키명 `teamver_design_bff_session` |
| D. FE sticky-decline 후 recovery 부족 | [00 07-13](./00_구현_내역_누적.md) | 401 후 toast | `recoverDriveAuthSession` |
| E. publish/import mutation recovery 부재 | [22 §3.2d](./22_Drive_인증_Usage_연동_검토.md) | mutation 401 | force refresh + FE cookie recovery |

본 문서의 **근본 해결 대상은 B**이다. A·C·D·E는 이미 보강됐지만, B가 남아 있으면 Drive는 계속 깨진다.

## 2. 근본 원인 — “마지막 Set-Cookie가 이긴다”

### 2.1 전제

- Design BFF session은 Starlette 서명 쿠키에 `{user_id, access_token, refresh_token, access_expires_at, …}`를 넣는다.
- Main Apps `/api/apps/auth/refresh`는 **refresh_token 1회 사용(회전)** 정책이다. 성공하면 이전 R0는 즉시 무효.
- design-api는 프로세스마다 in-memory refresh coalesce만 있다. **노드 간 공유 없음.**
- ALB sticky는 userId hash가 daemon 쪽 중심이고, **BFF/design-api cookie 경로는 형제 노드로 갈라질 수 있다.**

### 2.2 타임라인 (상세)

```
브라우저 Cookie: C0 = { access=A0, refresh=R0 }

t0  병렬 요청 R_a, R_b  (Drive뿐 아니라 /auth/session · workspace · bootstrap도 가능)
    ALB → Node A / Node B

t1  두 노드 모두 access skew 창 → Main refresh(R0) 시도

t2  Main: 먼저 도착한 Node A만 수락
        → { access=A1, refresh=R1 }
        Node A: save_bff_session → 응답 Set-Cookie: C1

t3  Node B: R0 이미 무효 → AppsTokenRefreshError
        access_token_is_usable(A0) == True  (아직 JWT headroom 있음)
        → 세션을 clear하지 않고 **C0 내용 그대로 retain**
        → (구) middleware가 매 응답 Set-Cookie → Set-Cookie: C0 재발행

t4  브라우저: 나중 도착 응답의 Set-Cookie가 이김
        Node B가 늦으면 Cookie = C0 복귀 (R0는 이미 무효)

t5  이후 모든 노드: R0로 refresh 재시도 → 실패 반복
        A0 만료 시 clear_bff_session → 401 session_expired + login_url
```

### 2.3 왜 “Drive만”처럼 보였나

1. Drive 모달은 진입 시 folder/list/shared-drive를 한꺼번에 연다 → 실패가 **한꺼번에** 눈에 띔.
2. 정작 쿠키를 stale로 되돌리는 경합은 Drive **이전**의 병렬 BFF 호출에서도 이미 일어난다.
3. FE `enqueueDriveFetch`는 Drive 호출을 직렬화하지만, 오염된 쿠키를 **치유하지는** 못한다 (이미 R0 무효).

### 2.4 구코드가 왜 매번 Set-Cookie를 냈나

`TeamverSessionMiddleware`(Starlette `SessionMiddleware` 동형)는 `scope["session"]`이 truthy이면 **응답마다** 서명값을 다시 붙였다.  
핸들러가 세션을 안 건드려도 “있는 그대로” 재서명 → **의미상 동일하지만 타이밍상 sibling 쿠키를 덮을 수 있는 Set-Cookie**가 된다.

`suppress_session_cookie`는 Drive/upstream 401 경로 등 **일부**에서만 켜졌고, 정상 200 + retain(세션 유지) 경로의 대부분에는 적용되지 않았다.

## 3. 해결 (무엇을 바꿨나)

### 3.1 핵심 — 변경된 세션만 Set-Cookie (`TeamverSessionMiddleware`)

파일: `deploy/teamver/be/app/middleware/teamver_session.py`

| 조건 | 동작 |
|------|------|
| 진입 스냅숏(`json.dumps(sort_keys=True)`)과 응답 직전 세션이 **동일** | **Set-Cookie 생략** |
| `save_bff_session` / workspace update 등으로 **변경** | canonical 쿠키 재발행 |
| 세션이 **비워짐** (logout / hard clear) | canonical delete + legacy delete |
| 레거시 이름(`session`)으로만 로드됨 | canonical 발행 + **legacy expire** (migrate) |
| `teamver_suppress_session_cookie` scope 플래그 | 아무 Set-Cookie도 안 냄 |

효과: Node B가 C0를 retain해도 **응답에 C0를 다시 심지 않음** → Node A의 C1이 브라우저에 남음.

### 3.2 방어 — retain 경로에서 명시적 suppress (`bff_tokens.py`)

파일: `deploy/teamver/be/app/auth/bff_tokens.py` · `_refresh_bff_session_core`

아래에서는 세션을 지우지 않고 돌려보내기 전에 `suppress_session_cookie(request)`:

- refresh_token 없음 + access 아직 usable
- Main unreachable + 기존 세션 유지
- refresh HTTP 실패(전형적 rotation race) + access 아직 usable

의도: middleware 비교 로직과 중복돼도, **“이 응답은 쿠키를 건드릴 자격이 없다”**를 코드로 고정.

### 3.3 hard-clear / logout이 suppress보다 이겨야 함

파일: `deploy/teamver/be/app/auth/bff_session.py`, `routers/projects.py`

| 규칙 | 이유 |
|------|------|
| `clear_bff_session`은 suppress 플래그를 **pop** | 진짜 만료·logout은 delete `Set-Cookie`가 나가야 함 |
| publish/import는 `force_refresh is None`이어도 **세션이 남아 있을 때만** suppress | hard-clear 뒤 blanket suppress → 죽은 쿠키 잔존 방지 |

### 3.4 쿠키 이름 격리 (병행 원인, 필수 운영)

파일: `config.py` · `main.py` · `validate_deploy_env.sh`

- canonical: `DESIGN_BFF_SESSION_COOKIE_NAME=teamver_design_bff_session` (기본값 동일)
- hosted에서 `session` 금지 + **비어 있음 금지**
- legacy `session`은 Design 서명일 때만 1회 로드 후 canonical로 승격하고 **expire**

Main의 `.teamver.com` `session`과 Design host-only `session`이 섞이면, 격리 없이는 서명 실패 → 빈 session → 같은 `session_expired`로 보인다. 이중화와 별개지만 **반드시 함께** 맞춰야 한다.

### 3.5 FE / nginx (이미 들어간 방어층, 본 수정의 전제)

- nginx: `/teamver-bff/drive/`는 `auth_request` 없음 → Drive router가 refresh·Set-Cookie 소유 ([22 §2.1b](./22_Drive_인증_Usage_연동_검토.md))
- FE: Drive fetch queue + soft retry + `recoverDriveAuthSession`
- publish/import: `force_refresh_bff_session` + FE `withDesignBffCookieAuthRecovery`

이것만으로는 B(모든 응답 Set-Cookie)를 막지 못했고, **3.1이 결정타**다.

### 5.2 staging 스모크

1. embed 로그인 → DevTools Application에서 `teamver_design_bff_session` 존재 확인.
2. Drive 모달을 여러 번 여닫기: folder/list/shared-drive **최종 200**.
3. 병렬로 `/teamver-bff/auth/session` 등을 반복해도, **세션을 안 바꾼 응답**에는 `Set-Cookie: teamver_design_bff_session`이 **없어야** 함.

5. 로그아웃 후 `teamver_design_bff_session`·잔존 Design `session`이 expire되는지 확인. 재진입 시 옛 세션이 부활하면 안 됨.

### 5.3 여전히 401이면

1. 응답 body가 `session_expired`(BFF)인지 Main `Invalid token`인지 구분.
2. nginx가 `/teamver-bff/drive/` 전용 location인지 (`auth_request` 없는지).
3. 모든 target의 `DESIGN_BFF_SESSION_SECRET` / `COOKIE_NAME` 동일성.
4. Main refresh rotation 로그와 design-api `[bff] refresh failed` 로그 시각 비교.
5. 단일 target으로 잠시 축소해 이중화 의존 여부 확인.

## 8. 후속 근본 수정 — Main Drive는 **HS256만** 수용한다

### 8.1 진짜 원인 (§2 위에 하나 더 있었다)

§2의 stale Set-Cookie 경합을 모두 막고도 재로그인 직후부터 아래가 재현됐다:

```
GET /teamver-bff/auth/refresh              → 200 {"status":"ok","authenticated":true}
GET /teamver-bff/drive/api/drive/folder    → 401 {"detail":"session_expired", ...}
GET /teamver-bff/drive/api/v2/shared-drive → 401 {"detail":"session_expired", ...}
```

즉 **BFF refresh는 성공**, 그런데 Drive는 계속 401. §1~§7의 HA cookie fix로는 설명이 안 된다.

Main BE 소스를 직접 확인한 결과:

| Main 라우트 | 인증 dependency | 검증 |
|-------------|-----------------|------|
| `/api/drive/folder`, `/api/drive/*` | `JWTService.get_current_user` | `jwt.decode(token, JWT_SECRET_KEY, algorithms=[HS256])` |
| `/api/v2/shared-drive/*` | `JWTService.get_current_user` | 동일 (HS256) |
| `/api/asset/upload-request`, `/api/asset/upload-confirm` | `JWTService.get_current_user` | 동일 (HS256) |
| `/internal/apps/{app_key}/*` | `require_internal_apps_token_user_in_db` | HS256 **또는** RS256 Apps JWT |
| `/api/apps/auth/refresh`, `/api/apps/auth/exchange` | `X-Teamver-Internal-Api-Key` M2M | 서버간 |

- BFF session이 저장하는 access_token은 M12 exchange가 발급한 **RS256 Apps JWT**(`aud=teamver-design`).
- 이 토큰은 **`/internal/apps/*`에서만** 유효. `/api/drive/*`, `/api/asset/*`, `/api/v2/shared-drive/*` 등 사용자용 REST는 HS256 platform JWT 전용.
- 따라서 Design Drive proxy가 `Authorization: Bearer <Apps RS256>`로 Main Drive를 호출하면 **loop 394부터 늘 401 "Invalid token"**. `/auth/refresh`가 200이어도 access_token 자체가 Main Drive 인증기에 맞지 않는다.

HA 이전엔 이 문제도 있었지만 sticky·타이밍상 sporadic이라 다른 원인(§A~E)에 묻혔고, 이중화 이후 병렬 부하가 증가하며 100% 재현이 됐다.

### 8.2 해결 — 브라우저의 Main SSO 쿠키를 Main으로 그대로 전달

Design host(`*.teamver.com`)에는 이미 Main의 `teamver_access_token`(HS256) 쿠키가 parent-domain SSO로 존재한다. nginx가 `proxy_set_header Cookie $http_cookie`로 design-api에 넘겨주므로 request에서 그대로 읽어 Bearer로 전달하면 된다.

- `deploy/teamver/be/app/routers/drive.py`
  - `_resolve_drive_access_token(request, auth)`가 **(token, source)** 튜플을 반환.
  - `_read_main_sso_cookie(request)`가 `teamver_access_token` HS256 쿠키를 우선 반환(`source="main_cookie"`).
  - 쿠키 부재 시에만 BFF Apps JWT로 폴백(`source="bff"`) — 로컬 dev · 오구성 대비.
  - Main HS256 401 시 **Apps refresh를 시도하지 않는다.** auth-shaped이면 즉시 `session_expired`(재로그인), 그 외 401(폴더 ACL 등)은 원문 pass-through.

- `deploy/teamver/be/app/routers/projects.py`

### 8.4 권고 · 대안 비교 (SSOT)

**선택지·권고의 상세 SSOT는 [41 Design Drive 인증 계약 권고](./41_Design_Drive_인증_계약_권고.md)다.** 요약만 남긴다.

| 방안 | 판정 |
|------|------|
| **A. Main SSO 쿠키 forwarding (본 §8)** | **지금 SSOT.** Main 변경 0 · `*.teamver.com` SSO 계약 재사용 · ROI 최대 |
| B. Main Drive/Asset dual-auth (Apps RS256 수용) | 조건부 중기. 타 도메인 Apps·서버 단독 Drive·platform JWT RS256 전환 시 |
| C. HS256 secret을 Apps와 공유 | **금지** (15_2) |
| D. Design이 Apps→HS256 재발급 | **금지** (Design이 IdP가 됨) |

알고리즘을 “같게” 만드는 게 목표가 아니다. Drive=platform JWT, bootstrap=Apps JWT로 **라우트별 패밀리를 맞추는 것**이 목표다.

### 8.5 운영 전제

- `AUTH_COOKIE_ENABLED=1`, `AUTH_COOKIE_DOMAIN=.teamver.com` (Main), Design host가 그 자식 서브도메인.
- nginx는 `proxy_set_header Cookie $http_cookie`를 Drive location(`/teamver-bff/drive/`)에서도 유지 — 현재 conf에 이미 포함.
- 브라우저에 Main SSO 쿠키가 없는 상태(로그아웃/타 도메인)에서 embed가 Drive를 부르면 즉시 `session_expired` + Main login_url 반환 → FE가 재로그인 유도.
- 체크리스트: [41 §6](./41_Design_Drive_인증_계약_권고.md).
