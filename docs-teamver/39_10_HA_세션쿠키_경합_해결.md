# Teamver Design — HA 세션 쿠키 경합·Drive 401 근본 해결 (39_10)

**목적:** ALB 다중 노드 환경에서 stg-design.teamver.com 사용 중 Drive 계열 API가 반복적으로 `401 { "detail": "session_expired" }`를 반환하던 회귀의 근본 원인·수정 내역·검증 절차 기록.  
**관련:** [39_0](./39_0_Design_이중화_로드맵_개요.md) · [39_2](./39_2_ALB_nginx_라우팅_설계.md) · [39_5](./39_5_검증_체크리스트_FAQ.md) · [22](./22_Drive_인증_Usage_연동_검토.md)

## 1. 증상

- 이중화(HA) 배포 이후 `/teamver-bff/drive/api/drive/*` 계열 호출이 결국 아래 응답을 받고 로그인 화면으로 튕겨나감:

  ```json
  {
    "detail": "session_expired",
    "login_url": "https://stg.teamver.com/auth/signin?app_id=teamver-design&redirect_url=..."
  }
  ```

- 재로그인 직후에는 잘 동작하다가, 몇 분간 다양한 요청을 하고 나면 Drive 모달 진입 시점에 반복적으로 실패.
- 단일 노드(ACTIVE 하나)에서는 재현되지 않음. **다중 노드일 때만 발생.**

---

## 2. 근본 원인 — HA 세션 쿠키 경합

`TeamverSessionMiddleware`가 **모든 응답에** 세션 쿠키를 재발행(Set-Cookie)하고 있었다. 여기에 `_refresh_bff_session_core`의 rotation-race 유지 로직이 결합되면서, 브라우저 쿠키가 **오래된 refresh_token**으로 롤백되는 케이스가 존재했다.

### 2.1 시나리오

1. 브라우저 쿠키 `C0 = {access=A0, refresh=R0}` 상태로 여러 동시 요청 발생.
2. ALB(sticky OFF)가 요청을 Node A / Node B 로 분산.
3. 두 노드가 동시에 access token skew 창을 보고 `refresh_apps_tokens_with_main(R0)` 호출.
4. **Main이 R0로 첫 응답만 수락**하고 새 tokens `{A1, R1}` 반환. Node A는 `save_bff_session` → `Set-Cookie: C1` 전송.
5. Node B는 이미 R0를 소진했으므로 401 → `AppsTokenRefreshError` → `access_token_is_usable(session)` True (A0 아직 유효) → **old session 유지 → `Set-Cookie: C0` 재발행**.
6. 브라우저는 나중에 도착한 응답의 `Set-Cookie`가 이긴다. Node B 응답이 뒤에 오면 브라우저 쿠키가 **stale `C0`로 복귀** — R0는 이미 무효화된 상태.
7. 이후 요청은 계속 R0로 refresh를 시도하다 실패 → A0 만료 시점에 `clear_bff_session` → `session_expired` 응답.
8. Drive 모달은 여러 병렬 호출을 유발하므로 이 경합이 특히 자주 트리거된다.

### 2.2 왜 기존 방어가 불충분했나

- `apps/web/src/teamver/driveApi.ts`의 `enqueueDriveFetch`가 Drive 호출은 직렬화하지만, **Drive 외 요청**(예: `/auth/session`, workspace probe, bootstrap)은 여전히 병렬로 여러 노드에 뿌려진다. 이 경합이 발생하면 Drive 모달을 열기도 전에 이미 쿠키가 stale이다.
- 기존 drive router의 `suppress_session_cookie` 호출은 upstream 401 이후 경로에서만 유효. 대부분의 정상 200 응답 경로에서는 middleware가 매번 `scope["session"]`을 그대로 재서명해서 내보내고 있었다.

## 3. 수정 요약

### 3.1 `TeamverSessionMiddleware` — 세션이 **변경되었을 때만** `Set-Cookie` 발행 (`deploy/teamver/be/app/middleware/teamver_session.py`)

- 요청 진입 시점의 `loaded_session`을 안정적인 형태(`json.dumps(sort_keys=True)`)로 스냅숏.
- 응답 발행 직전에 현재 `scope["session"]`을 다시 직렬화해서 비교:
  - **변경 없음** → `Set-Cookie` 생략. 형제 노드가 방금 rotate한 쿠키가 무사히 브라우저에 남는다.
  - **변경 있음** (`save_bff_session` / `clear_bff_session` / `update_bff_workspace` 등) → 기존대로 새 쿠키 발행.
  - **세션 삭제** (was non-empty, now empty) → 기존대로 delete 쿠키 발행.
- **레거시 쿠키 마이그레이션 예외**: 레거시 이름(`session`)으로 로드된 세션은 read-only 핸들러라도 canonical 이름(`teamver_design_bff_session`)으로 최소 1회 발행해야 후속 요청이 새 이름을 사용한다.
- `SUPPRESS_SESSION_COOKIE_SCOPE_KEY` 플래그가 켜지면 발행을 완전히 억제 (기존 동작 유지).

### 3.2 `_refresh_bff_session_core` — retain 경로에서 명시적으로 `suppress_session_cookie` (`deploy/teamver/be/app/auth/bff_tokens.py`)

세션 middleware의 새 로직만으로도 회귀는 재현되지 않지만, refresh 실패로 session을 유지할 때 명시적으로 `suppress_session_cookie(request)`를 호출해 **의도를 코드로 못박고** 향후 middleware 변경에 대한 안전망을 둔다.

- `missing_refresh_token` + `access_token_is_usable` retain 경로.
- `teamver_unreachable` retain 경로.
- `teamver_http_error` + `access_token_is_usable` rotation-race retain 경로.

### 3.3 새 테스트

- `test_teamver_session_middleware.py`
  - `test_session_middleware_omits_set_cookie_when_session_unchanged` — 핵심 회귀 방지.
  - `test_session_middleware_reissues_cookie_when_session_mutated` — 정상 rotate 경로 커버.
  - `test_session_middleware_still_migrates_legacy_cookie_without_mutation` — 레거시 마이그레이션 예외 유지.
  - `test_session_middleware_deletes_cookie_when_session_cleared` — 로그아웃 경로 유지.
  - `test_session_middleware_suppresses_set_cookie_when_flag_set` — suppress 플래그 커버.
- `test_bff_token_refresh_coalesce.py`
  - `test_refresh_auth_failure_retains_session_when_access_still_valid`에 `teamver_suppress_session_cookie is True` assertion 추가.
  - `test_refresh_unreachable_suppresses_stale_cookie` 신규.

Staging 이중화 스모크 (Runbook [39_4](./39_4_배포_Terraform_운영_Runbook.md) 절차):

1. staging에 배포 후 embed 로그인.
2. Drive 모달을 여러 번 여닫으며 다음 관찰:
   - `curl -I` 로 임의의 `/teamver-bff/*` 요청 여러 개 병렬 발사 시, 모든 응답이 `Set-Cookie: teamver_design_bff_session=` 를 무조건 반환하지 않아야 함 (변경 없는 요청은 헤더 없음).
   - Drive 열기 → 항상 200 응답. 401 세션 만료가 발생하지 않아야 함.
3. `/api/v1/auth/metrics`의 `auth.refresh.failure` 상승은 정상(형제 노드 rotation race 흔적)이지만, **user-facing 401 session_expired는 관찰되지 않아야 한다.**
4. 브라우저 devtools → Application → Cookies에서 `teamver_design_bff_session` 값이 페이지 이동/새로고침 중에도 항상 최신 rotation 값을 유지하는지 확인.

## 5. 후속 고려

- 노드 간 refresh 결과 공유는 여전히 없다(프로세스-로컬 `_refresh_apps_tokens_coalesced`). 하지만 이제 **stale 쿠키가 브라우저를 오염시키지 않으므로** 문제로 이어지지 않는다. Redis 등 공용 캐시는 굳이 도입할 필요가 없다.
- Main BE `/api/apps/auth/refresh`가 refresh_token 단일 사용 정책을 유지하는 한 이 해법이 유효하다. Main이 정책을 바꿔 다중 사용을 허용하면 middleware 재발행 정책은 그대로 두어도 안전하지만, `_refresh_bff_session_core`의 retain 경로가 활성화되는 빈도는 감소한다.
- FE의 `enqueueDriveFetch` 직렬화는 정상 사용자 UX(“동시에 4번 요청”)를 완화하는 방어층으로 유지한다. 필요 시 이후 이슈가 없다면 순차 제한을 조금 완화해도 안전하다.
