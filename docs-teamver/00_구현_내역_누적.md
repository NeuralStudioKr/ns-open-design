#### 2. 사실관계 정리

##### 2.1 OD 의 로컬 `PDF로 내보내기` 가 Drive 로 못 가는 진짜 이유

OD 의 로컬 PDF 내보내기는 **사용자 OS 파일로 직접 저장**되고 JS 에 PDF Blob 이 **돌아오지 않는** 구조다.

- **데스크탑(Electron)**: `webContents.printToPDF()` → native `Save` 다이얼로그 → 파일로 직접 write (`ns-open-design/apps/desktop/src/main/pdf-export.ts`). renderer 에는 `{ ok: true, path: <savedPath> }` 만 전달.
- **브라우저**: `window.print()` popup → 사용자 OS 인쇄 다이얼로그. JS 에 결과 안 옴.
- **OD daemon (`apps/daemon/src/import-export-routes.ts`)**: `POST /api/projects/:id/export/pdf` endpoint 존재 — 그러나 `desktopPdfExporter` 가 등록된 desktop runtime 일 때만 작동. headless staging/prod daemon 에선 501 (`desktop PDF export is only available in the desktop runtime`).

즉 **Drive 에 자동 업로드하려면 서버가 PDF bytes 를 만들어야** 하는데, OD design-api 에 그런 인프라는 **없다**.

##### 2.2 메인 팀버 BE 의 기존 PDF 인프라

`ns-teamver-be` 는 이미 Playwright + Chromium 기반 PDF 인프라를 운영 중:

| 파일 | 역할 |
|---|---|
| `src/service/canvas_doc/export/pdf_runtime.py` | lazy Chromium singleton, Semaphore(`CANVAS_PDF_MAX_CONCURRENCY=2`), 2s 세마포어 대기, 외부 URL abort (SSRF 차단), 연속 3회 실패 시 chromium restart |
| `src/service/canvas_doc/export/pdf_exporter.py` | A4 18mm 마진, `wait_until="load"`, `page.pdf(prefer_css_page_size=True)`, 15s 단일 렌더 타임아웃 |
| `Dockerfile` | `PLAYWRIGHT_BROWSERS_PATH=/opt/ms-playwright` + `playwright install --with-deps chromium` 통합 |
| `requirements.txt` | `playwright==1.50.0` |
| `docs/61_10_Canvas_PDF_다운로드_설계.md` | 옵션 비교 (Playwright / WeasyPrint / LibreOffice / reportlab / 외부 API) + 동시성·CJK 폰트 정책 |
| router | `POST /api/v2/canvas-doc-session/{sid}/item/{aid}/export-pdf` — `rich_document` 전용 (slide artifact 가 아님) |

핵심: 인프라 자체는 잘 설계돼 있고, 메인 BE 의 IR(internal representation) 에 묶여있어 **OD 슬라이드 HTML 을 그대로 받지 못한다**. 재사용하려면 별도 endpoint 신설이 필요.

##### 2.3 "Chromium 이 꼭 있어야 하느냐 / 서버 부하" 의견의 진짜 근거

> 이슈는 정확히는 _Playwright_ 가 아니라 **Chromium 런타임 자체**. wrapper(Playwright / Puppeteer / CDP 직접 호출 / Headless Chrome CLI) 를 바꿔도 Chromium 의 메모리·이미지·콜드스타트 비용은 그대로다. 이 한 줄짜리 정정은 main BE SSOT [`61_20 §1.1`](https://github.com/NeuralStudioKr/ns-teamver-be/blob/staging/docs/61_20_Canvas_PDF_Chromium_%EC%9D%98%EC%A1%B4%EC%84%B1_%EB%8C%80%EC%95%88_%EA%B2%80%ED%86%A0.md) 가 다룸. Chromium 없는 엔진 후보 (WeasyPrint / wkhtmltopdf / PrinceXML / 외부 SaaS / 클라이언트 측) 의 캔버스 정합·운영 매트릭스도 같은 문서 §3 에 있다.

| 우려 | 사실 |
|---|---|
| 메모리 | Chromium 인스턴스당 ~150–300MB. 메인 BE 는 동시 2개로 제한 + lazy 라 평상시 0. |
| CPU 스파이크 | 세마포어로 흡수 — 초과 요청은 2s 대기 후 429. |
| 응답 지연 | 동기 응답 — 2–5초 통상, 15초 타임아웃. **사용자 클릭 후 대기** 가 UX 상 가장 큰 단점. |
| Docker 이미지 | `--with-deps chromium` 으로 ~300MB 추가. 빌드·배포 시간 증가. |
| 진짜 문제 | API 서버 메모리·CPU 를 PDF 렌더링이 점유 → 다른 endpoint 응답 지연 가능성. **PDF worker 가 메인 API 서버와 같은 노드에 있는 한 구조적 한계는 남는다**. |

##### 2.4 Playwright 외 대안 비교

OD 슬라이드는 **JS 가 layout/animation/chart 를 그림** — JS 미실행 렌더러는 슬라이드를 깨뜨린다. 그래서 결국 **Chromium-class 엔진**이 필요하다. 다만 **어디서·어떻게 띄울지** 는 분리 가능.

| 방안 | Chromium 위치 | 비용/부하 | 응답 | 비고 |
|---|---|---|---|---|
| **A. OD design-api 에 Playwright 직접 추가** | OD BE | OD Docker +300MB, 메모리 점유 | 동기 | 가장 빠른 구현, 운영 부담 동일 |
| **B. 메인 팀버 BE 의 internal endpoint 호출** | 메인 BE | OD 추가 0, 메인 BE 부하 ↑ | 동기 | **인프라 통합 최적**, 권장 1순위 |
| **C. PDF 전용 microservice** (ECS / Fargate task) | 별도 worker | per-PDF ~$0.005, 격리, cold start ~3s | 동기/비동기 | 운영 비용·복잡도 ↑ |
| **D. AWS Lambda + chromium-lambda layer** (`sparticuz/chromium-min` ~40MB) | Lambda | per-PDF ~$0.001, cold start 1–2s | 비동기 권장 | 자원 효율 최고, IaC 추가 작업 |
| **E. 외부 SaaS** (Browserless / DocRaptor / PDFShift) | 외부 | 월 구독 $50+/도메인 의존 | 동기 | **외부 데이터 송출 보안 검토 필수** |
| **F. 클라이언트 측 (`html2pdf`/`jsPDF`+`html2canvas`)** | 사용자 브라우저 | 서버 0 | 동기, 사용자 대기 | **품질 손해 큼** — 래스터, 텍스트 검색 불가, 폰트/벡터 깨짐, OD slide JS layout 이슈 |
| **G. WeasyPrint / wkhtmltopdf** | OD BE | 메모리 ↓ | 동기 | **JS 미실행 → OD 슬라이드 못 그림** ❌ |

##### 2.5 응답 모델 비교 (PDF 도입 시점에 결정)

| 모델 | UX | BE 부하 | 구현 복잡도 |
|---|---|---|---|
| 동기 (sync) | 클릭 → 2–15초 spinner → 토스트 | 연결 시간만큼 worker 점유 | 낮음 — 현재 HTML 발행과 동일 시그니처 |
| 비동기 (async) | 클릭 → 즉시 history 에 `processing` 행 → polling/SSE 로 ready 전환 | worker 분리 가능 → API 노드 보호 | 중간 — `publish_status="processing"` 상태 추가 + worker queue + FE polling |

비동기는 OD design-api 의 `DesignOutput.publish_status` 가 이미 `"ready"|"failed"` 만 가지므로 `"processing"` 만 추가하면 도입 자연스러움. history panel (loop 174) 이 polling 만 추가하면 UX 부드러움.

#### 3. 결정

> **PDF / PPTX Drive 발행은 별도 트랙으로 보류** — 이번 출시 범위 밖. 현재 HTML-only 발행 + `HTML 한 파일로 발행됩니다. PDF/ZIP 추출은 다운로드 메뉴에서 로컬 저장하세요.` 안내 (loop 174) 가 그대로 유효.

PPTX 는 사용자 메시지에서 명시적으로 제외 ("PPTX는 개발 비용상 일단 제외한다"). PDF 도 현 시점 보류 — 의사결정 부담이 크고, 위 옵션 중 어느 것을 골라도 인프라 변경이 동반된다.

#### 4. 보류 사유

1. **시급성**: 현재 사용자 흐름 (OD 슬라이드 → AI 어시스턴트 컨텍스트) 에 HTML 만으로 충분. PDF 가 곧장 막는 use-case 가 명확하지 않음.
2. **인프라 결정 부담**: 위 옵션 A~F 각각 trade-off 가 있고, **어디서 chromium 띄울지 / sync vs async** 두 축 모두 의사결정·운영 가시화 작업이 필요. 이번 출시 게이트 (Track A) 와 동시 진행 시 리스크 큼.
3. **로컬 PDF 다운로드 는 이미 가능**: OD desktop runtime 의 `PDF로 내보내기` 가 정상 동작. 사용자는 로컬에서 PDF 받은 뒤 메인 팀버 Drive 에 수동 업로드도 가능 (1회성). 자동화 가치가 검증되기 전엔 인프라 투자 보류가 합리적.

#### 5. 재검토 트리거 (이 때가 오면 docket 을 다시 연다)

- **(T1) AI 어시스턴트가 Drive 의 PDF 본문을 인덱싱하는 시나리오가 우선순위로 올라옴** — 메인 팀버에서 슬라이드 PDF 를 컨텍스트로 쓰는 use-case 가 명확해지면.
- **(T2) 사내 PDF 인프라 가용성이 결정됨** — 메인 팀버 BE 가 internal HTML→PDF endpoint 를 정식 운영 서비스로 노출하거나, 별도 PDF microservice 가 도입되면 옵션 B/C 가 거의 무료가 된다. **main BE [`61_20`](https://github.com/NeuralStudioKr/ns-teamver-be/blob/staging/docs/61_20_Canvas_PDF_Chromium_%EC%9D%98%EC%A1%B4%EC%84%B1_%EB%8C%80%EC%95%88_%EA%B2%80%ED%86%A0.md) 의 트리거 T1/T5 결과 — 즉 main BE 측에서 사이드카 분리나 엔진 교체가 일어나면 OD 도 자동으로 본 docket 재검토 대상**.
- **(T3) 사용자가 자주 "Drive 에 PDF 가 있어야 한다" 라고 리포트** — 분기 1–2건 이상.
- **(T4) PPTX 가 별도 트랙으로 정식 착수** — 그때 PDF 와 함께 묶어 인프라 1회 결정.
