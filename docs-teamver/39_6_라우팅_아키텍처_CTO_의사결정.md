## 1. CTO 한 페이지 요약

### 1.1 무엇을 하는가

| 항목 | 내용 |
|------|------|
| **목표** | 동시 user 증가 시 **daemon(AI/SSE/scratch) 용량 2×** + **EC2 1대 장애 시** 서비스 지속 |
| **방법** | ALB 뒤 **EC2 2대**, 각 EC2에 nginx + daemon. **userId hash**로 daemon 작업을 user별 **고정 EC2**에 배치 |
| **ALB 역할** | TLS 종료, **round-robin** (userId 분산 **하지 않음**) |
| **nginx 역할** | 세션→userId 추출, `/api/*` **hash routing**, peer EC2 daemon으로 cross-proxy |

### 1.2 왜 user마다 같은 EC2인가 (vs 부하분산)

두 가지 **다른 목적**이다.

| 목적 | 설명 |
|------|------|
| **Affinity (user → EC2 고정)** | scratch·SQLite·SSE는 **노드 로컬**. user 요청이 EC2를 바뀌면 502·유실. **정합성** 목적 |
| **이중화 (EC2 2대)** | **서로 다른 user**를 EC2-1/2에 나눠 **동시 AI 처리량↑**, 한 대 down 시 나머지로 **HA**. **용량·가용성** 목적 |

→ “user별 고정”과 “서버 2대”는 **함께** 써야 의미 있다. user 고정만 1대면 HA/용량 없음. 2대만 user 고정 없으면 scratch 깨짐.

### 1.3 ALB에서 userId 분산이 불가

- userId는 **nginx `auth_request` → design-api session-probe** 이후에만 존재
- ALB는 **HTTP 헤더 hash / userId consistent hash** 미지원 (LBCookie sticky만 가능 → 브라우저 단위, 다기기 불일치)

### 1.4 장기 방향 (CTO 승인 요청 요지)

| 시점 | 구조 | hash key | daemon DB |
|------|------|----------|-----------|
| **Phase 4 (지금)** | ALB → **EC2 nginx** → peer hash | **userId** | SQLite 노드별 + Litestream |
| **Phase 5 (장기)** | **동일** (중앙 nginx 없음) | userId 유지; **projectId hash는 제품 필요 시** | **Postgres DaemonDb** (Track B) |
| **연기** | — | **project 단위 부하분산** | — |

**projectId hash로 “바꾸면 된다”**는 것은 **라우팅 **골격**(ALB + EC2 nginx + hash upstream + peer)** 은 그대로 두고, **hash key만** projectId로 바꾸는 **옵션**을 뜻한다. Track B(Postgres)와는 **별 트랙** — sqlite/scratch affinity 압력을 근본적으로 줄이려면 Phase 5가 정석이다.

## 2. 현재 동작 — 요청 한 건의 전체 경로

### 2.1 컴포넌트

```text
[Browser]
    │ HTTPS
    ▼
[ALB]  TLS 종료, target group = EC2-1, EC2-2 (nginx :80)
    │ round-robin (stickiness OFF)
    ├──────────────────┬──────────────────
    ▼                  ▼
[EC2-1 nginx]      [EC2-2 nginx]
    │                  │
    │ auth_request     │
    │ session-probe    │
    │ → userId         │
    │                  │
    │ hash(userId)     │
    ▼                  ▼
 open_design_daemon_hashed upstream:
   127.0.0.1:7456  +  peer private IP :7456
    │                  │
    └────────┬─────────┘
             ▼
    [선택된 EC2의 open-design-daemon]
             │
             ├─ scratch/ (로컬 디스크)
             ├─ app.sqlite (로컬, Litestream→S3)
             └─ S3 sync-down/up (프로젝트 파일 SSOT)
```

### 2.2 경로별 upstream (같은 EC2 nginx 안)

| 경로 | upstream | 분산 | 이유 |
|------|----------|------|------|
| `/_nginx/health`, `/_next/static`, `/favicon` | `open_design_daemon_local` (本機) | ALB RR | stateless static, 빌드 동일 |
| `/teamver-bff/*` | 本機 design-api :16000 | ALB RR | stateless BFF, RDS SSOT |
| `/api/*` (세션) | `open_design_daemon_hashed` | **userId hash** | scratch·SSE·SQLite |
| `/api/projects/:id/preview/*` (iframe) | hashed (key=**projectId in URI**) | projectId hash | session cookie 없음 |

**코드 SSOT:** `deploy/teamver/devops/nginx/teamver-design-od-daemon-upstream.inc.conf`

### 2.3 userId hash 상세 (홀짝·끝자리 **없음**)

nginx `hash $key consistent` (ketama):

1. hash key 문자열 **전체**를 해시 ring에 매핑
2. upstream server 목록 `{ 127.0.0.1:7456, peer-A:7456, peer-B:7456, … }` 중 **하나** 선택
3. **동일 key → 항상 동일 server** (userId `"user-abc"` → 항상 EC2-2 등)

**hash key 우선순위:**

```text
1) $teamver_user_id     ← auth 성공 시 (대부분의 /api/*)
2) URI 내 projectId     ← preview iframe
3) $binary_remote_addr  ← public asset fallback
```

### 2.4 cross-proxy (ALB가 고른 EC2 ≠ 작업 EC2)

**정상 동작**이다.

```text
User A → ALB → EC2-1 nginx → hash(A) → EC2-2:7456 daemon  ← 작업은 EC2-2
User B → ALB → EC2-2 nginx → hash(B) → EC2-2:7456 daemon  ← local
```

- EC2-1 nginx: **얇은 L7** (auth, 헤더, hash, SSE timeout). VPC private IP hop (보통 ms).
- **AI CPU·scratch I/O·SSE**는 hash가 고른 **daemon EC2**에서 실행 → **user 간 부하 분산**은 여기서 발생.

### 2.5 peer 목록·보안

| 항목 | SSOT |
|------|------|
| peer nginx include | `/etc/nginx/teamver-design-od-daemon-peers.inc` (`conf.d` 금지) |
| 생성 스크립트 | `deploy/teamver/scripts/render_od_daemon_peers_nginx.sh` |
| daemon publish | multi-node 시 `OD_DOCKER_PUBLISH_HOST=0.0.0.0` (本機 :7456) |
| SG | EC2 app SG → peer **7456** (Terraform `ec2_instance_count > 1`) |

## 3. 이중화(2 EC2) 효과 — 숫자 감각

**단일 EC2:** User A,B,C,D AI 동시 → **1대 CPU·export 큐** 경쟁.

**2 EC2 + userId hash (user 4명, hash 균등 가정):**

```text
User A, C → EC2-1 daemon
User B, D → EC2-2 daemon
→ 동시 stream·export가 2노드에 분산 (soft cap ~2×)
```

**한 EC2 장애:**

- hash된 user → 살아 있는 노드로 **재요청** → S3 sync-down → **수초~수십 초** 지연 (SSE 재연결)
- Litestream·S3로 **데이터 복구** 가능 ([39_4 failover](./39_4_배포_Terraform_운영_Runbook.md))

## 4. ALB sticky vs userId hash (과거 논의 정리)

| | ALB LBCookie sticky | userId hash (현재 SSOT) |
|--|---------------------|-------------------------|
| 키 | 브라우저 쿠키 | **userId 문자열** |
| 다기기 동일 user | ❌ 기기마다 다른 노드 | ✅ 같은 노드 |
| 시크릿 창 | ❌ 다른 노드 | ✅ 같은 userId면 같은 노드 |
| ALB 설정 | stickiness ON | stickiness **OFF** |
| 구현 | Terraform TG만 | nginx hash + peer |

Phase 4 **SSOT = userId hash**. staging `alb_stickiness_enabled = false`.

## 5. 장기 로드맵 — 구조는 유지, 무엇이 바뀔 수 있는가

### 5.1 고정 (CTO 합의 제안)

- **ALB → EC2 colocated nginx → hash → local/peer daemon** — **중앙 nginx 추가하지 않음**
- **projectId 단위 부하분산·hash 전환** — **제품/Phase 5 검토 후** (현재 **Out of scope**)
- **Phase 5 Track B:** `OD_DAEMON_DB=postgres` — daemon 메타 **multi-writer** (분기급)

### 5.2 나중에 바꿀 수 있는 것 (골격 유지)

| 변경 | 트리거 | 골격 변경? |
|------|--------|------------|
| hash key: userId → **projectId** | 동일 deck **다인 동시 편집** 제품화 | ❌ nginx hash key·헤더만 |
| **Postgres DaemonDb** | sqlite 노드 분열·운영 부담 | ❌ EC2/nginx 동일, daemon env |
| export **worker 분리** (Phase 3) | export CPU 80%+ | △ daemon colocation 일부 완화 |
| EC2 3~4대 | DAU·동시 AI ↑ | ❌ peer list·hash upstream만 확장 |

### 5.3 projectId hash — “나중에”의 의미

**연기 확정:** project 단위로 user를 **의도적으로 다른 EC2**에 나누는 **부하분산 정책**은 당분간 하지 않는다.

**장기 옵션 (제품 필요 시):**

- hash key를 `$teamver_user_id` → **`$project_id`** (FE `X-Project-Id` 또는 URL/body에서 nginx map)
- **효과:** 같은 project를 보는 **여러 user**가 **같은 scratch**를 쓰게 할 수 있음 (동시 공편집 전제)
- **한계:** projectId가 모든 `/api/*` 요청에 **항상** 있지 않음 → stream 시작 시 project binding·헤더 전파 설계 필요
- **Track B 없이** projectId hash만으로 sqlite 전역 SSOT는 **해결되지 않음** ([39_3](./39_3_scratch_SQLite_SSE_제약.md))

→ **장기:** 지금 **구조(중앙 nginx 없음)** 유지 + (필요 시) **hash key 교체** + (병행) **Postgres DaemonDb** 가 로드맵.  
**심화 FAQ:** [39_7 scratch·DaemonDb 저장층](./39_7_scratch_DaemonDb_저장층_심층_FAQ.md) — B5만으로 scratch 이슈가 사라지지 않는 이유, OD scratch 제거 불가.

## 6. 아키텍처 대안 비교 — EC2 colocated nginx vs 중앙 nginx

| 기준 | **A. 현재 (EC2 nginx + peer hash)** ✅ 채택 | **B. 중앙 nginx 1대 (+ EC2 daemon only)** |
|------|---------------------------------------------|-------------------------------------------|
| **토폴로지** | ALB RR → **각 EC2 nginx** → hash → local/peer :7456 | ALB → **전용 nginx EC2/ASG** → hash → daemon EC2들 |
| **userId hash** | ✅ EC2 nginx `auth_request` 후 hash | ✅ 중앙 nginx에서 동일 |
| **cross-proxy hop** | ⚠️ ALB→“엉뚱한” EC2 nginx → peer daemon (1 hop) | ✅ 항상 중앙→올바른 daemon (1 hop, 더 직선) |
| **auth_request** | ✅ 이미 production 패턴 | ✅ 이전 가능 (중앙에서 Main BE / design-api probe) |
| **SSE 3600s·CSP·plugin preview** | ✅ 기존 include 그대로 | ⚠️ **모든** location 규칙 중앙으로 이전·운영 |
| **SPOF** | ALB + **EC2 1대 down = 50% capacity** | ALB + **중앙 nginx down = 전체** (nginx HA 또 필요) |
| **배포** | EC2별 `deploy.sh` + peers render | daemon EC2 + **중앙 nginx 별도** rolling |
| **Terraform** | `ec2_instance_count` 확장만 | nginx ASG, SG, (또는 NLB+nginx) **신규 스택** |
| **비용** | EC2 2대 (nginx+daemon colocated) | EC2 2+ daemon + **nginx 전용 1~2대** |
| **Phase 5 전환** | hash key·Postgres만 | 중앙 nginx 유지 또는 제거 재설계 |
| **1차 상용 적합성** | ✅ 변경 최소, 39 시리즈 SSOT | △ hop은 줄지만 **새 SPOF·운영면** |

**CTO 제안:** Phase 4~5는 **A 유지**. 트래픽·cross-proxy p99가 문제로 **측정**될 때만 B 재검토.

## 8. CTO 보고 · 의견 요청 (템플릿)

### 8.1 보고 (2분 버전)

> Design Phase 4로 **EC2 2대 + ALB** 이중화를 staging부터 검증 중이다.  
> daemon(AI/SSE/scratch)은 **userId hash**로 user마다 고정 EC2에 배치하고, **서로 다른 user**는 두 노드에 나뉘어 **동시 처리량과 HA**를 확보한다.  
> ALB는 userId 분산이 불가해 **EC2 nginx**에서 auth 후 hash한다. **중앙 nginx는 두지 않는다.**  
> **projectId hash·동일 project 다인 편집**은 다음 페이즈로 연기했다. 장기 정석은 **Postgres DaemonDb(Track B)** 이다.

### 8.2 승인 요청 항목

| # | 결정 | 제안 | 대안 |
|---|------|------|------|
| 1 | Phase 4 affinity | **userId hash**, ALB sticky OFF | ALB cookie only (다기기 불리) |
| 2 | nginx 위치 | **EC2 colocated** + peer cross-proxy | 중앙 nginx (§6 표) |
| 3 | projectId hash | **연기** | 즉시 projectId hash (FE//nginx 공수↑) |
| 4 | Phase 5 예산 | **Postgres DaemonDb** 분기 로드맵 (**2026-07-08 CTO 우선 착수**) | SQLite+hash 장기 유지 |
| 5 | staging → prod | staging 2 EC2 리허설 **후** prod flip | prod 직행 |

### 8.3 CTO에게 물어볼 질문

1. **HA RTO/RPO** — EC2 1대 fail 시 **수십 초 SSE 끊김 + S3 sync-down** 수용 가능한가?
2. **동시 AI 목표** — 2 EC2로 **~10~20 concurrent stream** soft cap 목표가 맞는가?
3. **Phase 5 (Postgres DaemonDb)** — 상용 **6개월 내** Track B 착수 여부·우선순위 (export worker vs DaemonDb)?
4. **동일 project 다인** — 제품 로드맵에 **real-time co-editing** 이 있으면 projectId hash 시점 재논의.
