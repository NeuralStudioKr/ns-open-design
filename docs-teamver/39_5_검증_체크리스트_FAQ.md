|---|----------|------|------|
| A1 | 1 user AI stream 5min+ | 끊김 없음 (idle ALB) | ☐ |

| A6 | 두 노드 모두 살아있을 때 `curl` 반복 (cookie 없이) | `X-OD-Node-Id` **양쪽 값 모두** 관찰 → target 균형 | ☐ |
| A7 | design-api `X-Design-Api-Node` 헤더 확인 | staging: 두 값 관찰 / cookie 세션 안에서는 동일 | ☐ |

```bash
# 각 EC2 — peer upstream (userId hash 크로스 노드)
sudo apt-get install -y awscli
cd ~/neural/ns-open-design/deploy/teamver
sudo bash scripts/render_od_daemon_peers_nginx.sh
cat /etc/nginx/teamver-design-od-daemon-peers.inc
# → cluster private IP :7456 N줄 (sorted, self 포함 — 양쪽 EC2 동일). 0 peer 이면:
#   (1) IAM ec2:DescribeInstances — terraform iam.tf app_ec2_peer_discovery apply
#   (2) node1·node2 태그 Service=teamver-design, Env=staging
#   (3) 수동: server 10.10.101.x:7456 … (아래 §3.1.1)
sudo nginx -t && sudo systemctl reload nginx
```

#### 3.1.1 peer 0개 트러블슈팅

| 원인 | 확인 | 조치 |
|------|------|------|
| **IAM** | `aws ec2 describe-instances …` → AccessDenied | `teamver-design` terraform apply (`iam.tf` peer discovery policy) |
| **태그** | node1(구 EC2)에 `Service`/`Env` 없음 | AWS 콘솔 또는 `terraform apply` 로 태그 동기화 |
| **임시** | 알고 있는 private IP | peers conf 수동 (cluster 전체 IP, sorted) + `OD_DOCKER_PUBLISH_HOST=0.0.0.0` + deploy |
| **aws CLI 없음** | `command -v aws` 빈 결과 | `sudo apt-get install -y awscli` — 없으면 peers stub만 → [§3.1.3](#313-nginx-no-servers-are-inside-upstream--_nginxhealth-404) |

**2노드 수동 예** (경로는 `conf.d` **금지** — upstream `{ include }` 전용; **양쪽 EC2 동일 파일**):

```bash
sudo tee /etc/nginx/teamver-design-od-daemon-peers.inc <<'EOF'
server 10.10.101.169:7456 max_fails=2 fail_timeout=10s;
server 10.10.101.198:7456 max_fails=2 fail_timeout=10s;
EOF
sudo rm -f /etc/nginx/conf.d/teamver-design-od-daemon-peers.inc.conf
# upstream inc 에 127.0.0.1:7456 in hash ring 없음 (open_design_daemon_local 만 loopback)
sudo nginx -t && sudo systemctl reload nginx
cd ~/neural/ns-open-design/deploy/teamver
grep -q '^OD_DOCKER_PUBLISH_HOST=0.0.0.0' .env.staging || echo 'OD_DOCKER_PUBLISH_HOST=0.0.0.0' >> .env.staging
bash deploy.sh --staging --rds
```

#### 3.1.2 `/api/*` 502 — hash peer `:7456` unreachable

**증상:** `GET /api/version` · `/api/health` → **502 Bad Gateway** (nginx error log: `connect() failed` to `10.10.101.x:7456`).

**원인:** hash upstream이 cluster **private IP** 로 프록시하는데 Docker publish가 **`127.0.0.1:7456` only** (`OD_DOCKER_PUBLISH_HOST` 기본값).

**해결 (node1·node2 각각):**

```bash
grep OD_DOCKER_PUBLISH_HOST ~/neural/ns-open-design/deploy/teamver/.env.staging
curl -fsS http://10.10.101.169:7456/api/health   # peer from node2
curl -fsS http://10.10.101.198:7456/api/health   # peer from node1
bash deploy.sh --staging --rds
```

`.env.staging` 에 `OD_DOCKER_PUBLISH_HOST=0.0.0.0` + daemon recreate 후 private IP curl **200** 확인. SG `:7456` self 는 terraform `security_groups.tf`.

#### 3.1.3 `nginx: no servers are inside upstream` · `/_nginx/health` 404

**관련 runbook:** [39_4 §10.11](./39_4_배포_Terraform_운영_Runbook.md#1011-nginx-alb-httpconf-적용-순서--함정-prodstaging)

**증상**

1. apply 직후:

```text
⚠️ aws CLI not found — skipping peer render (single-node assumed)
nginx: [emerg] no servers are inside upstream in …/teamver-design-od-daemon-upstream.inc.conf:37
nginx: configuration file /etc/nginx/nginx.conf test failed
```

2. `sites-enabled` 에는 `*.http.conf` 만 있는데:

```bash
curl -sS http://127.0.0.1/_nginx/health
# → 404 Not Found (nginx/1.18.0 Ubuntu)
```

**원인 (연쇄)**

| 단계 | 무슨 일 |
|------|---------|
| 1 | `apply_*_nginx_conf.sh` 가 peers stub(주석만) 설치 |
| 2 | `aws` CLI 없으면 peer render가 **server 줄을 안 씀** (구버전) / 또는 발견 실패 |
| 3 | `upstream open_design_daemon_hashed { include peers.inc }` → **빈 upstream** → `nginx -t` **실패** |
| 4 | **reload 안 됨** — 디스크 conf만 바뀜. 또는 `default` 만 제거된 채 구 conf 유지 |
| 5 | ALB/로컬 Host 없는 `/_nginx/health` → Ubuntu default 또는 미적용 → **404** → TG unhealthy |

**해결**

```bash
# 1) peers에 cluster private IP 전부 (sorted, self 포함) — 양쪽 EC2 동일
sudo tee /etc/nginx/teamver-design-od-daemon-peers.inc >/dev/null <<'EOF'
server 10.10.101.63:7456 max_fails=2 fail_timeout=10s;
server 10.10.101.138:7456 max_fails=2 fail_timeout=10s;
EOF
# staging 예시는 §3.1.1 IP 사용

# 2) default 잔존 확인
ls -la /etc/nginx/sites-enabled/   # default 있으면 제거
sudo rm -f /etc/nginx/sites-enabled/default

# 3) test + reload
sudo nginx -t && sudo systemctl reload nginx
curl -fsS http://127.0.0.1/_nginx/health   # ok

# 4) 이후 자동화
sudo apt-get install -y awscli
cd ~/neural/ns-open-design/deploy/teamver
sudo bash scripts/render_od_daemon_peers_nginx.sh
```

**예방**

- 신규/부트스트랩 EC2: **먼저 `awscli` 설치** 후 nginx apply.
- apply 후 `cat /etc/nginx/teamver-design-od-daemon-peers.inc` 에 `server …:7456` 있는지 확인 — 주석만이면 reload 하지 말고 수동/render.
- `conf.d/` 아래에 peers 두지 말 것.

#### A5 검증 예시 (userId hash — 동일 user → 동일 X-OD-Node-Id)

```bash
# 로그인 세션 (Cookie) 으로 /api/health 반복 — userId hash 이므로
# 브라우저·ALB round-robin 과 무관하게 같은 daemon 노드여야 함.
COOKIE=$(mktemp)
curl -sI -c "$COOKIE" -b "$COOKIE" https://design.teamver.com/api/health | grep -i x-od-node-id
for i in $(seq 1 20); do
  curl -sI -b "$COOKIE" https://design.teamver.com/api/health |
    awk 'tolower($1) == "x-od-node-id:" {print}'
done | sort -u
# → 1줄만 출력 (userId hash OK)

# 다른 user / 미인증 — node id 가 달라질 수 있음 (정상)
rm "$COOKIE"
```

Cookie 없이 반복하면 (A6) 두 노드 값이 섞여 나와야 hash + RR 이 동작 중임이 증명된다.

## 5. FAQ

### Q1. 상용화 = 바로 이중화?

**아니다.** 오픈 1호기는 **Phase 0** ([39_0 §3](./39_0_Design_이중화_로드맵_개요.md)).  
HA SLA·DAU 성장 시 **Phase 2 또는 4**.

### Q2. 이중화하면 동시 AI가 2배?

**대략 2배에 가깝다** (2 EC2 AA, userId hash로 부하 분산) — 단 **ws당 8 cap은 노드별·전역 2× 자동 아님** ([38 §5](./38_Design_동시성_용량_확장_가이드.md)).  
soft cap: single ~5~10 → dual ~10~20 stream.

### Q3. userId sticky vs ALB cookie?

| | ALB cookie | nginx userId hash |
|--|------------|-------------------|
| 구현 | TG 설정만 | nginx upstream |
| 다기기 | 다른 노드 | **같은 userId → same node** |
| 미인증 | N/A | hash key empty 위험 |

**현재 SSOT (Phase 4):** nginx **userId hash**, ALB stickiness **OFF** ([39_6](./39_6_라우팅_아키텍처_CTO_의사결정.md)). ALB cookie는 다기기 불일치로 **비권장**.

### Q4. design-api도 2대?

**권장** — stateless, sticky OFF. daemon과 **같은 EC2** (colocated) 또는 별도 ASG.

### Q5. sticky 후에도 scratch 502?

→ [39_3 §4](./39_3_scratch_SQLite_SSE_제약.md) 동일 project 다인 · sync-up race · Main BE 502.  
→ **Phase 5** 또는 공편집 제한.

### Q6. Staging에서 Phase 4 테스트?

Staging tfstate 에 ALB + 2번째 EC2 를 확장해 리허설 — 자세한 절차는 [39_4 §7](./39_4_배포_Terraform_운영_Runbook.md) D1~D8. DNS/ACM 이관은 GCP Cloud DNS 이므로 수동. staging pass 후에만 prod tfvars `ec2_instance_count=2` flip.

### Q7. Phase 2 vs Phase 4 먼저?

| 목표 | 선택 |
|------|------|
| **HA만** (용량 동일) | **Phase 2** |
| **용량 2×** | **Phase 4** |
| **둘 다** | Phase 2로 runbook → Phase 4 |

### Q8. export worker (Phase 3) vs AA (Phase 4)?

export CPU **80%+** → **Phase 3 먼저**.  
AI latency·접속 수 → **Phase 4**.

### Q9. Postgres DaemonDb면 scratch 이슈 사라지나? OD scratch 제거 가능?

**B5(Postgres)만:** sqlite 분열 해소 — **scratch split-brain은 projectId affinity까지 Track B 전체** 필요.  
**OD scratch 제거:** 현재 아키텍처상 **불가**(agent CWD·Hybrid). 상세: **[39_7 FAQ](./39_7_scratch_DaemonDb_저장층_심층_FAQ.md)**.

### Q10. Embed 홈에서 프로젝트 목록이 비거나, 열면 404·빈 대화?

**원인:** Phase 4 이전 FE가 daemon sqlite 목록 + BFF registry **교집합**을 썼고, sqlite/대화는 **노드 로컬**이라 multi-node에서 불일치.

**코드 SSOT (staging `staging` 브랜치, Track B 전 브릿지):**

| 영역 | SSOT |
|------|------|
| Embed 목록·홈 recent | `teamver-bff/projects` (RDS) |
| `/api/projects/:id/**` 라우팅 | nginx **projectId hash** (`teamver-design-od-daemon-upstream.inc.conf`) |
| sqlite row 없을 때 | daemon registry hydrate (`teamver-project-sqlite-hydrate.ts`) |
| 대화·메시지 | S3 tenant `_daemon/project-state.v1.json` sync (Track B Postgres 전 임시) |

**검증:** 홈 Network — `teamver-bff/projects` 1회, `GET /api/projects?limit=24` 없음. 프로젝트 열기 후 `listConversations` 응답에 기존 thread 유지.  
**장기:** Postgres DaemonDb(Track B) — [39_7](./39_7_scratch_DaemonDb_저장층_심층_FAQ.md).

### Q11. apply nginx 후 health 404 · `no servers are inside upstream`?

**A:** peers.inc 가 비어 `nginx -t` 실패 → reload 안 됨 + (또는) Ubuntu `default` 잔존.  
→ [§3.1.3](#313-nginx-no-servers-are-inside-upstream--_nginxhealth-404) · [39_4 §10.11](./39_4_배포_Terraform_운영_Runbook.md#1011-nginx-alb-httpconf-적용-순서--함정-prodstaging).

### Q12. daemon이 `database "teamver_design_daemon_…" does not exist` 로 Restarting?

**A:** `OD_DAEMON_DB=postgres` 인데 RDS에 DaemonDb를 **CREATE 안 함**. design-api DB(`teamver_design_production`)와 **별도**.  
→ [39_9](./39_9_DaemonDb_B5_잔여_plugins_후속_및_RDS.md) `terraform output -raw rds_create_daemon_database_sql` 1회 실행 후 daemon 재기동.

### Q13. RDS 비밀번호가 갑자기 안 맞음 (pgAdmin도 실패)?

**A:** terraform plan에 `aws_db_instance … ~ password` 가 보이면 apply 시 **Enter a value / TF_VAR** 가 master를 덮음 (`ignore_changes` 없음). 오타면 옛 `.env`/pgAdmin 전부 실패.  
→ AWS master 재설정 후 env 동기화. 다음 plan에 password change 있으면 의도 확인 전 apply 금지.
