## 3. Phase 4 (Active-Active) — Affinity 검증

### 3.1 Functional

| # | 시나리오 | 기대 | Pass |
|---|----------|------|------|
| A1 | 1 user AI stream 5min+ | 끊김 없음 (idle ALB) | ☐ |
| A2 | 1 user stream 중 `GET /api/files` | 200, scratch 일치 | ☐ |
| A3 | 2 user **different ws** 동시 AI | 둘 다 완료 | ☐ |
| A4 | 1 ws **8 tab** 동시 AI | 8 OK, 9th **429** | ☐ |
| A5 | User sticky: `X-OD-Node-Id` 헤더 동일 노드 반복 확인 (20 requests, cookie 유지) | 모든 응답 **같은 node id** | ☐ |
| A6 | 두 노드 모두 살아있을 때 `curl` 반복 (cookie 없이) | `X-OD-Node-Id` **양쪽 값 모두** 관찰 → target 균형 | ☐ |
| A7 | design-api `X-Design-Api-Node` 헤더 확인 | staging: 두 값 관찰 / cookie 세션 안에서는 동일 | ☐ |

#### A5 검증 예시 (userId hash — 동일 user → 동일 X-OD-Node-Id)

```bash
# 로그인 세션 (Cookie) 으로 /api/health 반복 — userId hash 이므로
# 브라우저·ALB round-robin 과 무관하게 같은 daemon 노드여야 함.
COOKIE=$(mktemp)
curl -sI -c "$COOKIE" -b "$COOKIE" https://design.teamver.com/api/health | grep -i x-od-node-id
for i in $(seq 1 20); do
  curl -sI -b "$COOKIE" https://design.teamver.com/api/health |
    awk 'tolower($1) == "x-od-node-id:" {print}'
done | sort -u
# → 1줄만 출력 (userId hash OK)

# 다른 user / 미인증 — node id 가 달라질 수 있음 (정상)
rm "$COOKIE"
```

Cookie 없이 반복하면 (A6) 두 노드 값이 섞여 나와야 hash + RR 이 동작 중임이 증명된다.

### 3.2 Negative (sticky 없으면 실패하는 케이스)

| # | 시나리오 | sticky ON 기대 |
|---|----------|----------------|
| N1 | (테스트 only) stickiness OFF | stream+PUT mismatch → 502 **재현** |
| N2 | drain target during SSE | reconnect 또는 graceful error |

### 3.3 Failover (AA)

| # | 시나리오 | 기대 |
|---|----------|------|
| H1 | target-1 **강제 stop** | traffic → target-2 ≤ 60s, `X-OD-Node-Id` 값 flip 관찰 |
| H2 | H1 중 sticky user refresh | sync-down, slide 복구, 새 노드로 sticky 재고정 |
| H3 | `rolling_deploy.sh --env staging --dry-run` 통과 | 두 host 순서·drain·register 계획 그대로 출력 |
| H4 | `rolling_deploy.sh` 실제 실행 (staging) | zero **hard** outage (soft SSE blip 허용) |

## 5. FAQ

### Q1. 상용화 = 바로 이중화?

**아니다.** 오픈 1호기는 **Phase 0** ([39_0 §3](./39_0_Design_이중화_로드맵_개요.md)).  
HA SLA·DAU 성장 시 **Phase 2 또는 4**.

### Q2. 이중화하면 동시 AI가 2배?

**대략 2배에 가깝다** (2 EC2 AA, userId hash로 부하 분산) — 단 **ws당 8 cap은 노드별·전역 2× 자동 아님** ([38 §5](./38_Design_동시성_용량_확장_가이드.md)).  
soft cap: single ~5~10 → dual ~10~20 stream.

### Q3. userId sticky vs ALB cookie?

| | ALB cookie | nginx userId hash |
|--|------------|-------------------|
| 구현 | TG 설정만 | nginx upstream |
| 다기기 | 다른 노드 | **같은 userId → same node** |
| 미인증 | N/A | hash key empty 위험 |

**현재 SSOT (Phase 4):** nginx **userId hash**, ALB stickiness **OFF** ([39_6](./39_6_라우팅_아키텍처_CTO_의사결정.md)). ALB cookie는 다기기 불일치로 **비권장**.

### Q4. design-api도 2대?

**권장** — stateless, sticky OFF. daemon과 **같은 EC2** (colocated) 또는 별도 ASG.

### Q5. sticky 후에도 scratch 502?

→ [39_3 §4](./39_3_scratch_SQLite_SSE_제약.md) 동일 project 다인 · sync-up race · Main BE 502.  
→ **Phase 5** 또는 공편집 제한.

### Q6. Staging에서 Phase 4 테스트?

Staging tfstate 에 ALB + 2번째 EC2 를 확장해 리허설 — 자세한 절차는 [39_4 §7](./39_4_배포_Terraform_운영_Runbook.md) D1~D8. DNS/ACM 이관은 GCP Cloud DNS 이므로 수동. staging pass 후에만 prod tfvars `ec2_instance_count=2` flip.

### Q7. Phase 2 vs Phase 4 먼저?

| 목표 | 선택 |
|------|------|
| **HA만** (용량 동일) | **Phase 2** |
| **용량 2×** | **Phase 4** |
| **둘 다** | Phase 2로 runbook → Phase 4 |

### Q8. export worker (Phase 3) vs AA (Phase 4)?

export CPU **80%+** → **Phase 3 먼저**.  
AI latency·접속 수 → **Phase 4**.

### Q9. Postgres DaemonDb면 scratch 이슈 사라지나? OD scratch 제거 가능?

**B5(Postgres)만:** sqlite 분열 해소 — **scratch split-brain은 projectId affinity까지 Track B 전체** 필요.  
**OD scratch 제거:** 현재 아키텍처상 **불가**(agent CWD·Hybrid). 상세: **[39_7 FAQ](./39_7_scratch_DaemonDb_저장층_심층_FAQ.md)**.
